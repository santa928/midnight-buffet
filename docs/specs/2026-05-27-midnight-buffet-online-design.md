# ごちそう合戦 - Midnight Buffet オンライン祝宴版 設計仕様

- 作成日: 2026-05-27
- 対象: Vercel + Supabase による合言葉ルーム型オンライン版
- 前提: GitHub Pages 初期公開版の遊び、世界観、ルール判定を維持する
- ステータス: ユーザー承認済み（幹事進行 / 匿名認証 / 招待コード + 合言葉 / RLS + RPC）

## 1. 目的

会社や親戚の集まりで、それぞれのスマートフォンから同じ祝宴へ参加できるオンライン版を追加する。初期版の「端末を渡す」楽しさを残しつつ、参加者は自分の手元で予約札を秘密に封蝋し、幹事がクロッシュを開ける瞬間を全員が同時に見る。

オンライン版は初期版を置き換えない。開始画面で `この端末で遊ぶ` と `オンライン祝宴` を選択でき、通信環境や集まり方に合わせて両方を利用可能にする。

## 2. 設計決定

### 2.1 採用方式

Supabase Anonymous Auth、PostgreSQL Row Level Security (RLS)、Security Definer RPC、Realtime 更新通知を組み合わせる。Vercel は Vite アプリの配信を担当し、ルームの権限判定とゲーム状態更新は Supabase のトランザクション内で実行する。

### 2.2 採用理由

- 合言葉と公開前の予約札をブラウザへ広く配信せずに済む。
- 端末をリロードしても匿名セッションと参加記録から同じ席へ戻れる。
- ルーム状態をサーバー側で一貫して更新し、幹事端末の改変で勝敗が変わらない。
- 既存の React UI とゲーム用語を再利用しながら、オンライン固有の責務を adapter の外側へ分離できる。

### 2.3 不採用方式

| 方式 | 不採用理由 |
| --- | --- |
| Realtime Broadcast だけで端末間同期 | 状態復帰、秘密情報の保護、幹事切断時の一貫性が弱い。 |
| Vercel API が全ゲーム操作を仲介 | service key を扱う API 運用と同期処理が増え、初回オンライン版として過剰。 |

## 3. 世界観辞書の拡張

既存の `宴会ホール / ベルベット / 金の光 / クロッシュ / 予約札 / 封蝋 / 表彰` を維持し、オンラインの導線を劇場内の語彙で表現する。

| 機能語 | 表示語 | モチーフ |
| --- | --- | --- |
| room creation | 招待状を作る | 金縁の招待状、幹事の封印 |
| room join | 合言葉で入場 | 劇場入口、口頭の秘密 |
| lobby | 控室 | 参加者が席に着く待合空間 |
| host | 幹事 | クロッシュを開ける進行役 |
| waiting for bids | 封蝋待ち | 各席の封印済み印章だけを表示 |
| reconnect | 席へ戻る | 再入場し同じ席に復帰 |

オンライン画面は既存の宴会ホール背景、紺とベルベットのパネル、金縁ボタンを用いる。管理画面のような無機質な UI に切り替えない。

## 4. 体験フロー

### 4.1 幹事が招待状を作る

1. 幹事は `オンライン祝宴` から `招待状を作る` を選ぶ。
2. 表示名、`ショート9皿` / `フル15皿`、合言葉を入力する。
3. システムは Crockford Base32 のランダム10文字から成る招待コードを生成し、幹事を匿名認証して控室へ入れる。
4. 幹事は招待コードを画面やメッセージで共有し、合言葉は口頭で共有する。

### 4.2 招待客が入場する

1. 招待客は `合言葉で入場` を選び、招待コード、合言葉、表示名を入力する。
2. サーバー側 RPC がコードと合言葉を照合し、定員内かつ未開始なら席を作る。
3. 入場後は控室に参加者名だけを表示する。合言葉や認証 ID は表示しない。

### 4.3 幹事が開始する

1. 参加者が2〜6人になったら、幹事のみ `開宴する` を操作できる。
2. 開宴 RPC が料理順と全員の初期予約札を確定する。
3. 全端末は現在の料理と自分の予約札画面へ遷移する。

### 4.4 予約札を封蝋する

1. 各端末は本人の未使用予約札だけを表示する。
2. 本人は数字札を選び `この札を封蝋する` を押す。
3. 公開前に他の端末へ表示するのは `封蝋済み` の状態だけで、数字値は表示しない。
4. 全員が封蝋すると幹事に `クロッシュを開ける` が有効化される。

### 4.5 公開、次皿、表彰

1. 幹事が公開すると RPC が同値除外と獲得者判定を一度だけ実行する。
2. 公開後は全端末に数字札、衝突、獲得者、得点を表示する。
3. 幹事が `次の皿へ` を押すまで結果を鑑賞できる。
4. 最終皿後は表彰を全端末で表示し、幹事のみ同じ参加者での再戦を開始できる。

### 4.6 再接続

同じブラウザでページを再読込した場合、保持された匿名認証セッションから参加中のルームと席を取得し、現在のフェーズへ復帰する。別端末への席移譲や幹事権限移譲は初回オンライン版に含めない。

## 5. 権限と秘密情報

### 5.1 ブラウザへ公開してよい値

- `VITE_SUPABASE_URL`
- Supabase publishable key
- 招待コード
- 入場済みルームの公開状態、参加者表示名、公開後の結果

### 5.2 ブラウザへ公開しない値

- Supabase `service_role` key または secret key
- 合言葉の平文およびハッシュ
- 他プレイヤーの公開前の予約札
- 参加していないルームの状態

### 5.3 認証と参加権限

- 各端末は `signInAnonymously()` により Supabase Auth の `auth.uid()` を持つ。
- 参加者は1つのルーム内で1席だけを持つ。
- 幹事はルーム作成者の `auth.uid()` として記録し、進行 RPC のみ実行可能とする。
- 招待コードだけでは入場できず、参加 RPC 内で合言葉検証を成功させる必要がある。
- ルームは作成から24時間で失効し、失効後は入場と進行を拒否する。失敗した入場試行は認証端末と招待コード単位で記録し、短時間の連続試行を拒否する。

## 6. データモデル

合言葉検証用テーブルは API に露出する `public` schema から分離する。以下の公開テーブルはいずれも RLS を有効化し、所属ルームの参加者だけが必要最小限の行を読める。

| テーブル | 主な列 | 責務 |
| --- | --- | --- |
| `public.rooms` | `id`, `invite_code`, `host_user_id`, `mode`, `phase`, `round_index`, `current_dish_id`, `revision`, `expires_at`, timestamps | 同期対象となるルーム進行状態と24時間の有効期限。 |
| `private.room_secrets` | `room_id`, `passphrase_hash` | RPC だけが参照する合言葉 verifier。API から直接 select させない。 |
| `private.join_attempts` | `room_id`, `user_id`, `attempted_at`, `accepted` | 入場 RPC が参照する短時間の連続失敗抑制記録。 |
| `public.room_members` | `id`, `room_id`, `user_id`, `display_name`, `seat_index`, `joined_at` | 席と幹事を含む参加者一覧。 |
| `public.room_dishes` | `room_id`, `round_index`, `dish_id`, `points` | 開宴時に確定する料理順。 |
| `public.room_bids` | `room_id`, `round_index`, `member_id`, `bid_value`, `sealed_at` | 各席の封蝋札。公開前の値は本人以外 select 不可。 |
| `public.room_round_results` | `room_id`, `round_index`, `outcome`, `revealed_at` | 公開後の判定結果と得点同期。 |

必須インデックスは `invite_code` の unique index、全外部キー列、`(room_id, user_id)`、`(room_id, round_index)`、`(room_id, round_index, member_id)` の一意制約とする。

## 7. RPC と整合性

ブラウザはテーブルへ自由なゲーム更新を書き込まず、次の RPC を呼ぶ。各進行 RPC はルーム行をロックし、`revision` とフェーズを検証して二重操作を拒否する。

| RPC | 呼出可能者 | 成功時の処理 |
| --- | --- | --- |
| `create_banquet_room` | 認証済み匿名ユーザー | ルームと合言葉 verifier と幹事席を作成し招待コードを返す。 |
| `join_banquet_room` | 認証済み匿名ユーザー | 期限、試行抑制、合言葉照合、定員・名前重複・未開始を検証して席を作成する。 |
| `start_banquet_room` | 幹事 | 2〜6席を検証し、料理順と開始フェーズを確定する。 |
| `seal_banquet_bid` | 本人 | 未使用札、現在ラウンド、未封蝋を検証して本人の札を保存する。 |
| `reveal_banquet_round` | 幹事 | 全席封蝋済みを検証し、同値除外、獲得、得点を原子的に確定する。 |
| `advance_banquet_round` | 幹事 | 公開済み結果から次皿または表彰フェーズへ進める。 |
| `rematch_banquet_room` | 幹事 | 同じ席を維持して新しい料理順と未使用札状態を作る。 |

`reveal_banquet_round` は既存 TypeScript domain と同じルール値を用いる SQL 実装となるため、同一 fixture に対して TypeScript テストと DB テストの双方で結果一致を確認する。

## 8. RLS と Realtime

### 8.1 RLS 方針

- `rooms`、`room_members`、`room_dishes`、`room_round_results` の select は当該ルームのメンバーに限定する。
- `room_bids` の select は、公開前は `member.user_id = auth.uid()` の本人行だけ、公開後は当該ルームメンバーへ許可する。
- ゲーム状態の insert / update / delete は直接許可せず、権限を絞った Security Definer RPC だけで実行する。
- RPC は `authenticated` role だけに execute を許可し、固定した `search_path` と入力検証を持たせる。`private` schema と内部テーブルへの直接権限は `anon` / `authenticated` から revoke する。
- RLS の `auth.uid()` は `(select auth.uid())` として記述し、ルーム ID、外部キー、参加者照合列にはインデックスを設ける。

### 8.2 同期方針

- クライアントは参加後、所属ルームの `rooms`、`room_members`、`room_round_results` の変化を Realtime で購読する。
- 予約札については公開前の値を購読 payload へ依存せず、封蝋状態を安全な snapshot RPC から取得する。
- Realtime 通知を受けたら最新 snapshot を再取得し、UI を server state へ収束させる。
- 通知切断時は再購読と手動再読み込み導線を表示し、進行操作の失敗を成功扱いにしない。

## 9. フロントエンド境界

| モジュール | 責務 |
| --- | --- |
| `online/types` | 招待、控室、公開 snapshot、本人の private hand、通信エラーの型。 |
| `online/supabaseClient` | 公開環境変数の読取と lazy Supabase client 生成。オフライン導線では読み込まない。 |
| `online/onlineRoomAdapter` | RPC、snapshot 取得、Realtime 購読、再接続を UI 契約へ変換する。 |
| `online/scenes` | 招待作成、入場、控室、オンライン封蝋、幹事進行、切断時メッセージ。 |
| `domain` | 既存ルールの参照実装と fixture。通信や Supabase へ依存しない。 |

Supabase SDK は `オンライン祝宴` 選択後に動的 import し、一台回しの初期 bundle に不要な通信コードを載せない。

## 10. エラー処理

| 状況 | 表示と処理 |
| --- | --- |
| 招待コードまたは合言葉が不一致 | `招待状を確認してください` と表示し、どちらが誤りかは明示しない。 |
| 短時間に入場を繰り返し失敗 | `少し待ってからもう一度お試しください` と表示し、照合を一時拒否する。 |
| 定員超過または開宴済み | 入場不可理由を表示し、作成画面へ戻れる。 |
| ルームの24時間有効期限切れ | `この招待状の宴は終了しました` と表示し、新しい招待状作成へ誘導する。 |
| 名前重複 | 控室に入る前に別の表示名を求める。 |
| 使用済み札、二重封蝋、フェーズずれ | サーバー拒否後に snapshot を再取得し、現在状態を表示する。 |
| 幹事以外の進行操作 | UI にボタンを出さず、RPC でも拒否する。 |
| Realtime 切断 | `接続を確かめて席へ戻る` を表示し、再接続を試行する。 |
| リロード後に匿名セッションが失われた | 新規参加扱いとし、既存席の乗っ取りは許可しない。 |

## 13. 受け入れ条件

### 13.1 機能

- 幹事が部屋を作り、1〜5人の招待客がコードと合言葉で入場し、2〜6人で開宴できる。
- ランダム10文字の招待コードが生成され、失敗試行の抑制と24時間失効が動作する。
- ショート9皿とフル15皿で、各端末からの封蝋、幹事公開、得点反映、表彰、再戦を完走できる。
- オフライン版の一台回し導線と既存の完走テストが退行しない。
- 同じブラウザの再読込後に参加中の席へ復帰できる。

### 13.2 セキュリティ

- publishable key 以外の Supabase secret は client build、GitHub、Vercel 公開環境へ含まれない。
- 合言葉 verifier は API から直接 select できず、誤入力時に情報を過剰開示しない。
- 公開前の他人の札を select、Realtime payload、snapshot RPC のいずれからも取得できない。
- 幹事以外による開始、公開、次皿、再戦 RPC は拒否される。

### 13.3 UI と品質

- `390x844` と `430x932` の縦 viewport で、オンライン開始、控室、封蝋、公開、再接続メッセージの主要 CTA が操作可能で、料理舞台や HUD に重ならない。
- 控室から表彰まで、宴会ホール、招待状、クロッシュ、封蝋、金の光という世界観辞書を維持する。
- 接続待ちや操作拒否の状態を文言で通知し、色だけで状態を示さない。

### 13.4 配備

- Docker 内で frontend test / lint / build / Playwright と、ローカル Supabase の migration / RLS テストを実行できる。
- Vercel preview で2端末 E2E を通した後に、本番 URL へ昇格する。
- Supabase project 作成、migration 適用、Vercel 本番配備は本番反映として実行前に許可を得る。

## 14. 非対象

- ユーザー登録を伴う恒久アカウント。
- 公開ルーム一覧、観戦、チャット、絵文字リアクション、途中参加。
- 参加済み席の別端末移譲、幹事権限移譲。
- ランキング保存、課金、管理者ダッシュボード。

## 15. リスクと対策

| リスク | 対策 |
| --- | --- |
| 合言葉や札が client から漏れる | private secrets、RLS、RPC、publishable key のみの client 設定、秘密非露出テストを必須にする。 |
| 端末遅延で公開が二重実行される | ルーム行ロック、phase / revision 検査、原子的な公開 RPC で一度だけ結果を確定する。 |
| Realtime 切断で画面が古くなる | 通知は更新契機のみとし、操作前後に server snapshot を再取得する。 |
| Supabase SDK 追加でオフライン初期表示が重くなる | オンライン選択時だけ dynamic import する。 |
| Anonymous Auth の濫用 | ランダム招待コード、合言葉、定員、入場失敗試行抑制を実装し、攻撃兆候が確認された場合は CAPTCHA を追加する。 |

## 16. 性能目標

- オフライン開始画面の初期 bundle に Supabase SDK を含めず、公開済み体験の初期応答を大きく悪化させない。
- 正常な接続下で封蝋操作から控室への状態反映を1秒程度以内、幹事公開から結果同期を2秒程度以内の体感目標とする。
- ルーム取得は所属ルームと現ラウンドに限定し、全履歴の再取得を通常更新経路にしない。

## 17. 実装前検証方針

- Supabase 公式仕様に従い Anonymous Auth、publishable key、RLS、Realtime 購読 API の現行形を実装時に再確認する。
- migration は RLS を初回から有効化し、secret table を public API の対象に含めない。
- TypeScript domain fixture と SQL 公開判定の fixture を揃え、ルール差異を検出する。
